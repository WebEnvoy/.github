# WebEnvoy Vision

本文描述 WebEnvoy 作为完整产品体系的组织级愿景。

各仓库的 `VISION.md` 描述对应仓库的局部愿景；本文描述 WebEnvoy 整体要交付的用户价值、产品边界、系统分工和长期原则。

## 一句话愿景

WebEnvoy 是人类用户、Agent、自动化程序和上游系统使用真实网站能力的身份、上下文、能力、结果与观测基础设施。

它让网站读写任务在持续浏览器身份、低噪音上下文、可复用能力资产、归一化读写结果和可观测运行记录之上，变成可执行、可验证、可恢复、可对账、可迭代的稳定能力。

更短地说：

> WebEnvoy 让人类用户、Agent 和上游系统以持续身份和低噪音上下文复用真实网站能力，并把读写过程、结果和知识资产沉淀为可观测、可验证、可迭代的系统事实。

## WebEnvoy 要解决的问题

WebEnvoy 面向 Agent，但不只服务 Agent。

真实网站能力的使用者包括：

- 人类用户：管理任务、能力资产、账号环境、运行现场、异常接管和修复流程；
- Agent：调用网站能力、完成读写任务、消费低噪音上下文和结构化结果；
- 自动化程序：把网站能力接入脚本、批处理和本地执行流程；
- 上游系统：通过 API、SDK、CLI、MCP 或 App 调用网站能力并消费结果；
- 开发者和团队：创建、修复、测试、版本管理和分发能力资产。

这些使用者遇到的问题不是“能不能打开网页”，而是能否稳定、低噪音、低风控、可验证、可恢复地使用真实网站能力。

WebEnvoy 要解决真实网站能力使用中的九个断点。

## 九个断点

### 1. 身份与低风控断点

真实网站不是无状态 API。

一个账号能否稳定使用，取决于账号、Profile、Cookie、storage、代理、地区、语言、时区、浏览器指纹、登录历史、使用习惯和风险状态组成的连续身份。

没有 WebEnvoy 时，真实网站任务经常出现：

- 登录态不连续；
- 浏览器环境变化突兀；
- Profile 与账号绑定不清；
- 代理、地区、语言和时区不一致；
- 自动化会话像临时拼装的客户端；
- 验证码、登录异常、访问受限和风控状态难以及时识别；
- 任务在资源或状态不满足时继续硬跑。

WebEnvoy 的目标不是承诺绕过所有风控，也不是把反检测能力完全外包给某个浏览器 provider。

WebEnvoy 要追求的产品结果是：让人类用户、Agent 和自动化程序在网站侧尽可能表现为稳定、连续、合理的正常用户，并在出现风险信号时识别、停止、接管和记录。

这要求：

- Harbor 维护 Profile、执行身份、浏览器环境、provider 能力事实和运行现场；
- Lode 为能力声明资源需求、账号要求、浏览器上下文要求和风控状态识别方式；
- WebEnvoy Core 在执行前做资源匹配，在执行中识别异常状态，在资源或状态不满足时停止；
- WebEnvoy App 让人类用户查看身份状态、环境状态、风控状态、接管入口和恢复流程。

### 2. 上下文与噪声断点

网页现场通常包含大量低价值上下文。

没有 WebEnvoy 时，调用方经常需要面对：

- 截图；
- 完整 DOM；
- 冗长 HTML；
- 原始接口响应；
- 网络噪音；
- 临时日志；
- 无关页面元素；
- 重复探索历史；
- 不稳定的现场推理。

对 Agent 来说，这是 token 成本和推理噪声。对上游系统来说，这是接口不稳定和结果不可消费。对人类用户来说，这是排查成本和认知负担。

WebEnvoy 应把复杂网页现场压缩成低噪音、结构化、可消费的上下文：

- 站点知识；
- 能力入口；
- Snapshot / RefMap；
- 页面状态摘要；
- 资源需求；
- 结构化失败原因；
- 证据引用；
- result envelope。

WebEnvoy 不应让 Agent 每次用大量 token 重新理解网页，也不应让上游系统直接消费高噪音现场材料。

WebEnvoy 的产品价值之一，是让网页任务从高噪音现场探索，变成低噪音能力调用。

### 3. 能力复用断点

很多网站操作不是一次性点击，而是可复用流程。

没有 WebEnvoy 时，站点入口、页面状态、操作路径、接口字段、资源要求、登录判断、风控页识别、上传流程、发布流程和验证逻辑经常散落在脚本、提示词、截图和临时调试记录中。

WebEnvoy 应把这些经验沉淀为结构化能力资产：

- 站点知识；
- 站点能力；
- 原子动作；
- 任务封装；
- 模板；
- 测试样例；
- 资源需求；
- 输入输出契约；
- 版本和失效标记。

人类用户、Agent、自动化程序和上游系统应优先复用已经沉淀的站点能力和任务封装，而不是每次重新探索同一个网站。

### 4. 执行策略与资源匹配断点

WebEnvoy 不应该只是“有能力就执行”。

真实网站任务需要在执行前回答：

- 这个能力是否已经准入；
- 当前输入是否有效；
- 当前账号是否满足任务要求；
- 当前 Profile 是否满足任务要求；
- 是否需要完整浏览器上下文；
- 是否允许页面内请求或外部 API；
- 是否必须有头执行；
- 是否需要写入前检查和写入后验证；
- provider 是否具备所需能力事实；
- 当前资源不足时是否应该停止。

WebEnvoy 不只执行动作，也要在能力、资源、身份、站点状态和运行现场之间做准入与匹配。

Lode 声明资源需求和能力边界；Harbor 提供 provider、Profile 和 Runtime Session 的客观能力事实；WebEnvoy Core 负责能力准入、资源匹配、任务执行和停止条件；WebEnvoy App 负责让人类用户理解当前能力和资源状态。

### 5. 知识资产迭代断点

WebEnvoy 不应该是一组一次性脚本。

网站会变化，页面会改版，接口会变，字段会消失，风控页会出现，资源需求会升级，能力会失效。

没有 WebEnvoy 时，一次成功探索、一次失败修复、一次页面变化识别，往往不能反哺系统。

WebEnvoy 应形成知识资产迭代闭环：

```text
探索
  → 固化能力
  → 执行任务
  → 记录证据
  → 失败归因
  → 识别站点变化
  → 生成修复草稿
  → 测试
  → 版本更新
  → 用户 overlay 或平台资产演化
```

每次成功执行、失败修复、页面变化、字段变化、风控状态识别，都应尽可能反哺 Lode 资产。

WebEnvoy 的长期价值不是保存一次经验，而是让网站知识和能力资产持续演化。

### 6. 读写结果断点

WebEnvoy 不应只把网站任务结果作为 raw payload、截图、DOM 片段、临时字段或一句“成功 / 失败”返回。

读侧任务需要归一化结果，让不同网站、不同能力版本、不同执行通道产生的数据可以被一致消费。

读侧结果应支持表达：

- content item；
- author / profile item；
- comment item；
- media item；
- metric / counter item；
- dataset record；
- dedup_key；
- raw_payload_ref；
- evidence_ref；
- source_trace；
- captured_at；
- 字段缺失、字段不可见、字段被隐藏或采集失败的结构化原因。

写侧任务也需要结构化结果，而不是模糊成功状态。

写侧结果应支持表达：

- write_operation_ref；
- idempotency_key；
- target_object_ref；
- draft_id；
- published_url；
- object_id；
- pre_check_result；
- post_check_result；
- verification_result；
- unknown_outcome；
- manual_recovery_required；
- reconcile_status；
- evidence_ref。

WebEnvoy 不应成为通用数据仓库或离线 ETL 平台，但它必须让网站读写结果具备稳定、可校验、可追溯、可消费、可对账的公共形态。

Lode 负责定义输出契约、fixture 和测试样例；WebEnvoy Core 负责运行时校验、投影、封装和记录；Harbor 负责提供 raw reference、source trace、Snapshot、网络摘要和执行证据；WebEnvoy App 负责展示、导出和辅助用户理解这些结果。

### 7. 任务可观测断点

没有 WebEnvoy 时，网页自动化任务经常是黑盒。

用户和上游系统难以回答：

- 任务是否真的执行了；
- 用了哪个能力版本；
- 用了哪个账号、Profile 和 Runtime Session；
- 资源是否满足；
- 当前状态是什么；
- 在哪一步失败；
- 失败是网站变化、能力失效、账号异常、风控、资源不足、运行时异常还是输入错误；
- 写入是否真的成功；
- 证据在哪里；
- 是否需要人工恢复；
- 后续能否对账。

WebEnvoy 应让网页任务从黑盒自动化变成可观察过程。

任务运行应形成 durable truth，而不是只留下临时日志。WebEnvoy Core 应维护 Run Record，记录任务请求、能力版本、资源匹配结果、Runtime Session、执行身份引用、步骤状态、写入前检查、写入后验证、失败分类、Evidence 引用、unknown outcome、manual recovery 和 reconcile status。

WebEnvoy 的产品价值之一，是让任务可观察、可恢复、可审计、可对账。

### 8. 人机协作与控制权断点

真实网站任务不可能永远全自动。

验证码、登录异常、权限提示、文件选择、复杂页面、页面卡住、写入结果不确定等情况，都可能需要人类介入。

WebEnvoy 应把人类用户作为一等参与者，而不是只把人类视为配置者或旁观者。

人类用户应能够：

- 查看运行现场；
- 暂停任务；
- 人工接管；
- 处理异常；
- 恢复任务；
- 修复能力；
- 确认写入；
- 查看证据；
- 提交失效上报。

WebEnvoy 是人类、Agent、自动化程序和上游系统共同使用真实网站能力的基础设施。Agent 是主要场景之一，但不是唯一使用者。

### 9. 隐私边界与资产治理断点

WebEnvoy 会处理敏感状态和敏感结果。

这包括：

- 账号；
- Cookie；
- token；
- Profile；
- 浏览器现场；
- 用户业务输入；
- 写入内容；
- 原始请求响应；
- 截图；
- DOM；
- 运行证据；
- 用户个人能力资产。

因此，组织级产品边界必须明确。

默认原则是：

- 真实账号动作默认在用户环境中执行；
- 默认不上传账号、Cookie、token、完整 DOM、完整请求响应、用户业务内容或未脱敏执行现场；
- 公共上报应是脱敏站点变化摘要和能力失效事实；
- 平台资产和用户个人资产必须区分；
- 用户 overlay、fork 和 draft 默认不污染平台资产；
- 公共能力和公共模板需要审核后分发；
- 用户个人资产默认私有，但可以导出、同步或可选提交为平台贡献。

隐私边界和资产治理不是附属规则，而是 WebEnvoy 获得用户信任的产品基础。

## WebEnvoy 交付的核心价值

WebEnvoy 的整体价值不是某一个模块单独完成的，而是 App、Core、Harbor 和 Lode 协作形成的产品结果。

长期来看，WebEnvoy 交付以下核心价值：

1. **持续浏览器身份与低风控执行**：让真实网站任务基于连续身份和一致环境执行，并能识别风险状态、停止错误执行和支持人工接管。
2. **低噪音上下文与更低 token 成本**：把网页现场压缩为结构化上下文和能力调用，减少截图、DOM、日志和重复探索带来的噪音。
3. **可复用站点能力与任务封装**：把网站入口、状态、动作、验证和资源需求沉淀为可版本化能力资产。
4. **执行准入与资源匹配**：在能力、身份、Profile、provider、浏览器上下文、站点状态和用户策略之间做可解释匹配。
5. **读写结果归一化与状态对账**：让采集、发布、修改、删除和提交结果具备稳定、可校验、可追溯、可消费、可对账的公共形态。
6. **任务可观测与运行证据**：让每次任务有 Run Record、Evidence、失败分类、恢复入口和对账状态。
7. **知识资产持续演化**：让成功经验、失败修复、站点变化和风控识别持续反哺能力资产。
8. **人机协作与恢复**：让人类用户能够观察、暂停、接管、修复、确认和恢复任务。
9. **隐私边界与资产治理**：区分平台资产和用户个人资产，默认保护账号、会话、业务输入和执行现场。

## 目标产品形态

WebEnvoy 的长期产品形态包括：

```text
WebEnvoy App
  统一人类用户入口

WebEnvoy Core
  统一任务执行路径、能力准入、资源匹配、Run Record、结果封装和失败归因

Harbor
  浏览器身份、Profile、Runtime Session、Viewer、人工接管、provider 能力事实和运行证据

Lode
  站点知识、站点能力、原子动作、任务封装、模板、测试样例、输出契约、资源需求和版本资产
```

整体协作关系：

```text
WebEnvoy App
  ├── Work    → WebEnvoy Core
  ├── Library → Lode
  └── Browser → Harbor

Agent / SDK / CLI / MCP / 自动化程序 / 上游系统
  → WebEnvoy API Server
  → WebEnvoy Core Runtime
  → Lode 能力资产
  → Harbor Runtime Session
  → Run Record / Result Envelope / Evidence
```

WebEnvoy App 是人类用户入口，不是 WebEnvoy 的唯一产品形态。Agent、自动化程序、SDK、CLI、MCP 和上游系统应通过 WebEnvoy Core 使用网站能力。

## 核心仓库分工

### WebEnvoy/App

`WebEnvoy/App` 是统一人类用户入口。

它承载三个产品域：

```text
Work    → 任务、运行记录、结果、证据、恢复
Library → 能力资产、模板、探索、版本、失效上报
Browser → 浏览器身份、Profile、Runtime Session、Viewer、人工接管
```

App 是产品入口，不是任务执行真相源，也不是能力资产真相源。

### WebEnvoy/WebEnvoy

`WebEnvoy/WebEnvoy` 是 WebEnvoy Core 仓库。

Core 负责：

- API Server；
- Core Runtime；
- 能力准入；
- 资源需求匹配；
- 任务执行路径；
- Run Record；
- result envelope；
- 结果归一；
- 失败归因；
- 写入安全；
- 写入对账；
- SDK、CLI、MCP 等入口复用同一执行路径。

Core 不是完整 WebEnvoy 产品体系，也不承载统一 App Shell。

### WebEnvoy/Harbor

`WebEnvoy/Harbor` 是 Agent-ready Profile Runtime。

Harbor 负责：

- Profile；
- Execution Identity；
- Runtime Session；
- Browser Drivers；
- CDP / VNC；
- Viewer；
- 人工接管；
- provider 能力事实；
- 运行证据；
- Runtime API。

Harbor 不理解具体网站业务，不解释站点结果，也不定义 normalized result。

### WebEnvoy/Lode

`WebEnvoy/Lode` 是能力资产库。

Lode 负责：

- 站点知识；
- 站点能力；
- 原子动作；
- 任务封装；
- 模板；
- 测试样例；
- 输出契约；
- 结果 schema；
- 资源需求；
- 版本；
- 失效标记；
- 平台资产与用户 overlay / fork / draft 的边界。

Lode 不管理 Runtime、Profile、CDP、VNC、用户账号或真实运行现场。

### WebEnvoy/research

`WebEnvoy/research` 保存研究材料、外部项目调研、横向对比、技术主题和决策候选。

research 不作为正式产品文档或 API Contract 的真相源。

### WebEnvoy/.github

`WebEnvoy/.github` 保存组织级文档、仓库地图、许可证边界、issue / PR 模板和协作规则。

组织级 `VISION.md` 位于本仓库。

## 端到端工作流

一个典型任务应沿着以下路径运行：

```text
用户、Agent、自动化程序或上游系统提交任务
  → WebEnvoy API Server
  → Core Runtime 接收任务
  → 读取 Lode 能力声明
  → 检查能力是否准入
  → 校验输入和资源需求
  → 向 Harbor 请求 Runtime Session / Capability Facts
  → Core 判断资源是否满足
  → 执行能力或任务封装
  → Harbor 提供浏览器现场和运行证据
  → Core 写入 Run Record
  → Core 校验和封装结果
  → 返回 Result Envelope / Evidence / Failure Reason
```

如果任务涉及写入：

```text
写入任务
  → 写入前检查
  → 幂等控制
  → 执行写入
  → 写入后验证
  → 成功 / 失败 / unknown_outcome / manual_recovery_required
  → 后续对账
```

如果能力失效或网站变化：

```text
任务失败
  → 记录失败分类和证据引用
  → 标记能力或用户 overlay 可能失效
  → 进入 App / Library 修复流程
  → 形成能力草稿或修复草稿
  → 测试
  → 保存为用户个人资产
  → 可选提交脱敏平台上报
```

## 长期产品原则

WebEnvoy 长期应坚持以下原则：

1. WebEnvoy 面向 Agent，但不只服务 Agent。
2. WebEnvoy 是真实网站能力基础设施，不是通用浏览器 Agent。
3. 能力优先，而不是浏览器优先。
4. 持续浏览器身份、隐身性和低风控执行是组织级产品目标。
5. 不承诺绕过所有风控，但必须维护身份连续性、环境一致性、行为合理性和风险状态识别。
6. 低噪音上下文是核心价值，不能让调用方默认消费完整现场噪音。
7. 真实读写都是一等能力。
8. 真实写入必须可检查、可验证、可对账。
9. 所有任务入口应复用同一核心执行路径。
10. 未准入能力不进入稳定执行路径。
11. 资源不满足时不硬跑。
12. Profile 不是最高层语义，执行身份才是。
13. 越靠近账号、写入和风控，越应维护浏览器上下文和身份连续性。
14. 读写结果必须有稳定、可校验、可追溯的公共形态。
15. Run Record 是 durable truth，不是临时日志。
16. 任务应可观察、可恢复、可审计、可对账。
17. 人类观察和接管是一等能力。
18. Lode 资产应可版本化、可测试、可修复、可回滚。
19. 知识资产应随成功经验、失败修复和站点变化持续迭代。
20. 平台资产和用户个人资产必须清楚区分。
21. 用户 overlay、fork 和 draft 默认不污染平台资产。
22. 公共能力和公共模板需要审核后分发。
23. 默认不上传账号、Cookie、token、完整 DOM、完整请求响应、用户业务内容或未脱敏执行现场。
24. WebEnvoy 返回结构化事实，不输出不可解释的黑盒评分。
25. WebEnvoy 不替用户决定业务策略。
26. 底层浏览器 provider 可以替换，但上层能力、身份和证据模型应保持稳定。
27. 产品上统一，架构上分离。

## WebEnvoy 不是什么

WebEnvoy 不是通用浏览器 Agent。

它不应该只靠截图、DOM 和模型现场推理完成任务；正式任务路径应优先复用已准入的站点能力和任务封装。

WebEnvoy 不是单纯的指纹浏览器。

它需要持续浏览器身份、隐身性和低风控执行能力，但目标不是只管理 Profile、代理和指纹，而是让这些身份服务于网站能力执行、结果归一和任务可观测。

WebEnvoy 不是普通爬虫或数据仓库。

它需要采集数据归一化和读侧结果契约，但不承担通用离线 ETL、数据湖或业务数据仓库职责。

WebEnvoy 不是业务策略系统。

它不决定用户应该发布什么、采集什么、联系谁、如何运营账号或如何制定业务策略。

WebEnvoy 不是账号矩阵运营系统。

它可以管理执行身份和浏览器环境，但不负责账号增长、内容排期、广告投放、CRM 或运营决策。

WebEnvoy 负责的是：

> 当用户、Agent、自动化程序或上游系统已经知道要完成什么网站任务时，让这个任务通过持续身份、低噪音上下文、可复用能力、归一化结果和可观测运行记录稳定执行出来。

## 成功状态

当 WebEnvoy 成功时，用户、Agent、自动化程序和上游系统应该可以自然地说：

- 我不需要每次重新摸索同一个网站；
- Agent 不需要消费大量截图、DOM 和现场噪音；
- 网站经验已经沉淀成可安装、可版本化、可测试、可修复的能力资产；
- 我可以通过统一入口运行任务、查看结果、恢复异常和管理证据；
- 任务执行前，系统会检查能力准入、输入和资源需求；
- 资源不满足时，系统会停止并返回明确原因；
- 网站任务基于长期维护的浏览器身份执行，而不是一次性自动化会话；
- 系统能识别验证码、登录异常、访问受限和其他风险状态；
- 任务卡住时，人类可以看到现场并接管；
- 写入任务有前置检查、后置验证和状态对账；
- 读侧结果有稳定、可追溯、可消费的归一化形态；
- 写侧结果不是模糊成功状态，而是可验证、可恢复、可对账的运行事实；
- 失败后，我能判断是网站变化、能力失效、账号异常、资源不足、运行时异常、风控状态还是输入错误；
- 我能看到 Run Record、Evidence、资源匹配事实和能力版本；
- 我可以更新、锁定、回滚或修复能力资产；
- 我的个人能力修改不会污染平台资产；
- 公共能力可以通过脱敏上报持续进化；
- WebEnvoy 返回的是结构化事实，而不是不可解释的黑盒判断。

这就是 WebEnvoy 的长期产品价值：让真实网站能力在持续身份、低噪音上下文、能力资产、归一化结果和任务可观测之上，成为可复用、可验证、可恢复、可对账、可迭代的系统能力。
